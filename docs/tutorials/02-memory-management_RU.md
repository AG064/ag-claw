# Урок 2: Управление памятью

*Примерное время: 20 минут*

AG-Claw имеет одну из самых сложных систем памяти среди всех фреймворков для агентов. Этот урок объясняет, как она работает и как использовать её для создания агентов, которые запоминают, учатся и улучшаются со временем.

---

## Четыре уровня памяти

AG-Claw использует четыре различных системы памяти, каждая оптимизирована для своей цели:

```
┌─────────────────────────────────────────────────────────────────┐
│  Уровень 1: Семантическая память  (быстрая, поисковая, обильная)│
│  Хранит: разговоры, решения, уроки, ошибки, предпочтения       │
│  Backend: SQLite + эмбеддинги                                   │
├─────────────────────────────────────────────────────────────────┤
│  Уровень 2: Граф знаний           (структурированный, связный)   │
│  Хранит: сущности, связи, факты                                 │
│  Backend: SQLite с обходом графа                               │
├─────────────────────────────────────────────────────────────────┤
│  Уровень 3: Markdown-память       (читаемая человеком, стойкая) │
│  Хранит: заметки, документацию, долгосрочные факты              │
│  Backend: Файлы на диске, отслеживаемые на изменения           │
├─────────────────────────────────────────────────────────────────┤
│  Уровень 4: Саморазвивающаяся память (сжатая, абстрактная)      │
│  Хранит: консолидированные сводки старых воспоминаний            │
│  Backend: Автоматический процесс консолидации                   │
└─────────────────────────────────────────────────────────────────┘
```

---

## Уровень 1: Семантическая память

Основная рабочая память. Каждое взаимодействие, решение, урок и ошибка сохраняются здесь.

### Типы памяти

| Тип | Когда используется | Пример |
|---|---|---|
| `decision` | Вы сделали выбор | "Мы решили использовать PostgreSQL" |
| `lesson` | Вы узнали что-то ценное | "Никогда не запускай `rm -rf /` на продакшен-сервере" |
| `error` | Что-то пошло не так, и вы это исправили | "API возвращал 500 из-за отсутствующего индекса" |
| `preference` | Личное предпочтение пользователя | "Алексей предпочитает Rust языку Go" |
| `general` | Всё остальное worth запоминания | Заметки со встреч, факты, контекст |

### Автозахват

AG-Claw автоматически обнаруживает и сохраняет важную информацию из ваших сообщений. Когда вы говорите что-то вроде:

- "Запомни, что..." → сохраняется как `general`
- "Мы решили..." → сохраняется как `decision`
- "Я узнал, что..." → сохраняется как `lesson`
- "Баг был вызван..." → сохраняется как `error`
- "Я предпочитаю..." → сохраняется как `preference`

Функция автозахвата автоматически анализирует ваши сообщения и извлекает эти паттерны.

### Ручные операции с памятью

```bash
# Сохранить что-то вручную
agclaw memory store "decision" "Использовать feature flags вместо веток для постепенного развёртывания"

# Поиск в памяти
agclaw memory search "feature flags"

# Просмотр недавних воспоминаний
agclaw memory recent --limit 10

# Просмотр статистики памяти
agclaw memory stats
# Вывод:
# Всего записей: 1543
# По типам:
#   decision: 234
#   lesson: 567
#   error: 89
#   preference: 123
#   general: 530
```

### Использование памяти как инструмента агента

Агент может использовать инструменты памяти напрямую во время разговора:

```
Пользователь: С какими ошибками мы сталкивались в API?
Агент: [вызывает memory_search с запросом="ошибки API"]
     Найдено 2 релевантные записи:
     [error] API возвращал 500 из-за отсутствующего индекса в колонке user_id (2026-03-20)
     [error] Rate limiter блокировал легитимные запросы после последнего деплоя (2026-03-18)

Пользователь: Хорошо. И какие решения мы приняли по базе данных?
Агент: [вызывает memory_search с типом="decision", запросом="база данных"]
     Найдено 3 решения:
     [decision] Использовать PostgreSQL как основную базу данных (2026-03-15)
     [decision] Добавить Redis для кэширования данных сессий (2026-03-16)
     [decision] Запускать миграции базы данных при старте, не при деплое (2026-03-18)
```

## Уровень 2: Граф знаний

Граф знаний хранит сущности и их связи как графовую структуру. Это позволяет рассуждать о связях между фактами.

### Что сохраняется

- **Узлы**: Люди, проекты, компании, технологии, концепции
- **Рёбра**: Связи между узлами (`works_at`, `depends_on`, `uses`, `created_by`)

### Пример графа

```
(Пользователь) ──(создал)──► (Проект: AG-Claw)
   │                      │
   │                      ├──(использует)──► (Технология: TypeScript)
   │                      │
   │                      ├──(использует)──► (Технология: SQLite)
   │                      │
   │                      └──(разрабатывается)──► (Человек: AG064)
```

### Запросы к графу

```bash
# Просмотр статистики графа
agclaw memory graph stats
# Вывод:
# Узлов: 342
# Рёбер: 891
# Самый связный: AG-Claw (12 связей)

# Экспорт графа
agclaw memory graph export > graph.json
```

### Как граф обновляется

Когда семантическая память сохраняет новую запись, функция графа знаний анализирует её:

1. Извлекает именованные сущности (люди, места, технологии)
2. Определяет связи между сущностями
3. Создаёт или обновляет узлы и рёбра графа
4. Связывает новую информацию с существующими знаниями

Это означает, что граф становится умнее со временем по мере того, как агент обрабатывает больше информации.

## Уровень 3: Markdown-память

Markdown-память хранит заметки в читаемом формате как файлы `.md` на диске. Этот слой для информации, которую вы хотите:

- Хранить постоянно
- Редактировать вручную
- Версионировать в git
- Делиться с другими инструментами

### Как это работает

Создавайте или редактируйте файлы в директории `memory/`:

```bash
# Директория памяти
ls memory/

# memory/projects.md
# memory/people.md
# memory/notes.md
```

Пример `memory/projects.md`:

```markdown
# Проекты

## AG-Claw
- Тип: AI Agent Framework
- Статус: Активная разработка
- Лид: AG064
- Технологии: TypeScript, SQLite, Claude
- URL: https://github.com/AG064/ag-claw

## Личный сайт
- Тип: Статический сайт (Astro)
- Статус: Завершён
- URL: https://example.com
```

Функция markdown-memory отслеживает изменения этих файлов и автоматически обновляет контекст агента, когда соответствующие файлы изменяются.

### Интеграция с агентом

Агент может читать файлы markdown-памяти, когда они релевантны разговору:

```
Пользователь: Над какими проектами мы работаем?
Агент: [вызывает read_file на memory/projects.md]
     Согласно вашим заметкам о проектах, вы работаете над:
     1. AG-Claw — фреймворк AI-агента (активный)
     2. Личный сайт — статический сайт (завершён)
```

## Уровень 4: Саморазвивающаяся память

Со временем семантическая память накапливает тысячи записей. Функция саморазвивающейся памяти периодически консолидирует старые записи, чтобы система оставалась эффективной.

### Как работает консолидация

Когда память превышает `compressionThreshold` (по умолчанию: 10 000 записей):

1. **Анализ**: Находит кластеры связанных записей
2. **Абстракция**: Заменяет несколько конкретных записей обобщёнными сводками
3. **Прунинг**: Удаляет маловажные или избыточные записи
4. **Сохранение**: Сохраняет существенную информацию в сжатом виде

### Пример

До консолидации (12 отдельных записей):
- "Встреча с Бобом о дизайне API — используем REST" (1 марта)
- "Боб подтвердил, что REST — правильный выбор" (2 марта)
- "Обсудили REST с Бобом, согласовали префикс /api/v1" (3 марта)
- ... (ещё 9 похожих записей)

После консолидации (1 абстрактная запись):
- "Решили использовать REST API с префиксом /api/v1. Боб участвовал в решении." (1-15 марта)

### Настройка консолидации

```json
{
  "features": {
    "self-evolving-memory": {
      "enabled": true,
      "compressionThreshold": 10000,
      "consolidationIntervalHours": 24,
      "minEntriesBeforeConsolidation": 5000
    }
  }
}
```

### Система чекпоинтов

Для долгих задач используйте чекпоинты для сохранения прогресса:

```bash
# Агент вызывает memory_checkpoint во время сложной задачи
Агент: memory_checkpoint(taskId="refactor-auth-2026", state={
  "step": 3,
  "completed": ["login-form", "session-store"],
  "current": "token-validation",
  "remaining": ["refresh-token", "logout"]
})

# Позже возобновить из чекпоинта
Агент: memory_resume(taskId="refactor-auth-2026")
# Возвращает: { step: 3, completed: [...], current: "token-validation", ... }
```

## Эффективное использование памяти

### Лучшие практики

1. **Будьте явными при сохранении важной информации**
   Не полагайтесь на то, что автозахват что-то поймает:
   ```
   Пользователь: Запомни, что всегда нужно запускать тесты перед пушем.
   Агент: [вызывает memory_store с типом="lesson", content="Всегда запускай тесты перед пушем"]
   ```

2. **Используйте правильный тип памяти**
   Не храните всё как `general`. Используйте типы `decision`, `lesson`, `error`, `preference` — они делают поиск более точным.

3. **Запрашивайте на естественном языке**
   Семантический поиск понимает контекст:
   ```
   agclaw memory search "вещи которые пошли не так с миграцией базы данных"
   ```

4. **Позвольте графу строиться естественно**
   Вам не нужно вручную наполнять граф знаний. Он растёт по мере использования памяти — просто сохраняйте семантические записи, и функция графа извлечёт сущности и связи.

5. **Используйте markdown для постоянных знаний**
   Факты, которые не должны меняться (детали проекта, структура команды), принадлежат markdown-памяти, не семантической.

### Память в мультиагентных настройках

При запуске нескольких агентов память может быть общей или изолированной:

```json
{
  "agents": [
    {
      "id": "coding-assistant",
      "memory": { "shared": false, "namespace": "coding" }
    },
    {
      "id": "research-assistant",
      "memory": { "shared": false, "namespace": "research" }
    },
    {
      "id": "coordinator",
      "memory": { "shared": true }
    }
  ]
}
```

Координационный агент видит всю память, а специализированные агенты — только свою.

## Решение проблем с памятью

### Поиск в памяти не возвращает результатов

1. Проверьте, что `semantic-search` включён
2. Убедитесь, что записи существуют: `agclaw memory stats`
3. Попробуйте более широкий запрос: `agclaw memory search "и"`
4. Сохраните тестовую запись: `agclaw memory store "test" "testing memory"`

### Слишком много дублирующих записей

Функция консолидации должна с этим справляться. Если она не работает:
1. Проверьте, что `self-evolving-memory` включена
2. Убедитесь, что `compressionThreshold` настроен правильно
3. Вручную запустите консолидацию: проверьте здоровье функции

### Память замедляет агента

1. Уменьшите `messageHistory` в конфигурации webchat (меньше сообщений в памяти)
2. Увеличьте `compressionThreshold` для более ранней консолидации
3. Отключите неиспользуемые функции (они могут сохранять ненужные данные)
